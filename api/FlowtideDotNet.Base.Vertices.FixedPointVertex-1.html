Remarks

A fixed-point vertex coordinates two distinct input sources — an ingress input carrying new external data and a feedback input carrying data recycled from the loop — and routes processed results to two output sources: an egress output forwarding results downstream and a loop output feeding data back into the cycle. Derived classes express the per-message processing logic via OnIngressRecieve(T, long) and OnFeedbackRecieve(T, long), mapping each output event to index 0 (egress) or 1 (loop).

Checkpointing differs fundamentally from MultipleInputVertex<T>: when a checkpoint arrives on the ingress input, the vertex sends a FlowtideDotNet.Base.LockingEventPrepare into the loop and waits for the loop to confirm that no new messages are in flight before propagating the checkpoint to the egress output. The method NoReadSourceInLoop() controls this behaviour — returning true indicates no persistent read source exists in the loop, allowing the checkpoint to complete without waiting for an upstream acknowledgement.

NoReadSourceInLoop()

Returns whether there is any persistent read source present within the fixed-point loop.

public abstract bool NoReadSourceInLoop()

Returns

bool

true if no read source is present in the loop; otherwise false.

Remarks

This value directly affects checkpoint coordination. When true, the vertex does not wait for an upstream loop acknowledgement before propagating the checkpoint, because no external read source can produce additional messages mid-checkpoint. When false, the vertex waits until the loop has confirmed it is idle before forwarding the checkpoint to the egress output.
