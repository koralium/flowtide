# Distributed Mode

> [!WARNING]
> Distributed mode is still experimental.

Distributed mode splits one stream into multiple *substreams* that can run in different processes or on different machines. Each substream is a complete Flowtide stream with its own state storage, and the substreams exchange data with each other through *exchange operators*.

This differs from [Parallelism](../parallelism.md), which partitions operators inside a single stream. Parallelism helps when a single operator is the bottleneck on one machine, distributed mode helps when the whole stream no longer fits on one machine, either in memory, in state size, or in compute.

## How it works

* The plan is split so that partition-heavy operators such as joins, aggregates and window functions run with one partition copy in every substream. Data is scattered between the substreams by hashing on the partition keys, and the results are gathered back to the substream that runs the sink.
* Operator chains that stay on the same partition key, for example a join followed by a filter and an aggregate grouped on the join key, run entirely inside each substream, so only the final results cross substream boundaries.
* Data that crosses between substreams goes through transient queues. The receiving substream pulls the data, together with checkpoint barriers and watermarks, so watermarks and checkpoints work across substream boundaries exactly like inside one stream.

## Consistency and failure recovery

Substreams checkpoint together. A checkpoint cycle in one substream only completes when the substreams it exchanges data with have acknowledged it, so no substream can compact state that another substream might still need after a rollback.

When any substream fails, every substream it exchanges data with rolls back to a common checkpoint version and the data between the checkpoints is regenerated by replaying from the sources. This is the same recovery model as a single stream, extended across the substream group: sources must be able to re-send data from their last committed position, which is the standard requirement for any Flowtide source.

When substreams restart, an initialize handshake compares their restored checkpoint versions. If the versions differ, for example because one substream lost its state, all substreams recover to the lowest common version and catch up by replay.

## Hosting options

* [In-process hosting](inprocess.md) runs all substreams in one process. This is mainly useful for testing and for verifying how a plan distributes before deploying it.
* [Orleans hosting](orleans.md) runs each substream as an Orleans grain, letting the substreams spread across a silo cluster with automatic placement and recovery when silos fail.

## Controlling the distribution

* [Automatic distribution](automaticdistribution.md) takes a completely normal plan and splits it into a chosen number of substreams.
* [SQL substream statements](sqlsubstreams.md) give manual control over which statements run in which substream.

## Limitations

* Recursive queries (iterations) are never split across substreams, a plan containing an iteration keeps that whole pipeline inside one substream.
* Joins where no key is compared with equality cannot be partitioned and stay inside one substream.
* Aggregates that use the surrogate key function are not distributed.
* A distributed view consumed from another substream must declare `SCATTER_BY`, broadcasting across substreams is not supported and is rejected when the plan is built.
* Running the substreams on one machine is normally slower than running the same plan as a single stream, the exchange adds serialization and queueing per crossing batch. Distributed mode pays off when the stream needs more resources than one machine has.
